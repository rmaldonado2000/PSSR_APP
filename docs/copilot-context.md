# PSSR Code App - Copilot Context

## App Purpose

Digitally manages the Pre-Startup Safety Review (PSSR) process, including plans, checklists, deficiencies, approvals, and team roles, using Dataverse as the backend. The app enforces lifecycle rules and role-based access for each phase.

## Technology Stack

- React 19 with TypeScript
- Vite build tool
- Fluent UI React Components
- Power Apps Code App
- Dataverse CRUD via generated services
- ESLint
- Vitest

## Current App Structure

- `src/app/`: Core logic, lifecycle, Dataverse repository, types, routing, and i18n
- `src/components/`: Shared UI components such as `Pill`, `CardAccent`, and other reusable UI elements
- `src/generated/`: Autogenerated Dataverse models and services
- `src/screens/`: Main app screens including Plans, Plan Details, Checklist Details, and Template Library
- `src/ui/`: Design tokens such as pill tokens and visual standards
- `docs/`: Lifecycle, schema, gallery/card, pill, and Copilot context documentation

## Main Screens

- `PlansScreen`: Plan gallery, filtering, and navigation
- `PlanDetailsScreen`: Plan metadata, checklists, deficiencies, approvals, and team information
- `ChecklistDetailsScreen`: Checklist runner, question answering, and deficiencies
- `TemplateLibraryScreen`: Checklist and question templates with role-aware visibility

## Dataverse Integration

- Uses generated services for CRUD operations, including services such as `Crc07_pssr_plansService`, `Crc07_pssr_checklistsService`, `Crc07_pssr_checklist_questionsService`, and related generated service files
- Main tables include `crc07_pssr_plans`, `crc07_pssr_checklists`, `crc07_pssr_checklist_questions`, `crc07_pssr_deficiencies`, `crc07_pssr_approvals`, `crc07_pssr_team_members`, and `systemusers`
- Option sets for phases, statuses, roles, categories, and related choices are mapped in generated model files
- Data access is routed through the repository layer in `dataverseRepository.ts`

## Lifecycle / Status / Phase Rules

- Plan phases: Draft -> Plan -> Execution -> Approval -> Completion
- Checklist statuses: Not Started -> In Progress -> Completed
- Deficiency statuses: Open -> In Progress -> Closed
- Approval statuses include Approved, Rejected, Completed, and In Progress
- Lifecycle transitions and UI locks are enforced in `lifecycle.ts` and `lifecycleTransitions.ts`
- Full lifecycle schema and behavior are documented in `docs/pssr-lifecycle-schema.md` and `docs/pssr-lifecycle-behavior.md`

## UI / UX Standards

- Fluent UI is used for controls and layout
- Gallery/list cards follow the shared anatomy documented in `docs/ui-gallery-card-anatomy.md`
- Pills/badges follow the token and color standard documented in `docs/ui-pill-standard.md`
- Responsive design is required for mobile and desktop
- Pill and accent rendering uses `getPillStyle` and `getAccentColor` from `ui/tokens/pillTokens.ts`
- Styles are centralized through `index.css` and Fluent `makeStyles`

## Reusable Components

- `Pill`: Used for status, phase, and neutral pills with standardized visual rules
- `CardAccent`: Used for consistent card accent coloring
- `GalleryCard`, `RowCard`, `SectionPanel`, `VirtualizedList`, `DataState`, and related reusable UI components
- Shared components are imported through `components/ui.tsx`

## Current Known Constraints

- The app supports the currently generated Dataverse schema; Dataverse option set or table changes may require regeneration of generated models/services
- Plan phase changes are command-driven; manual phase editing is not part of the current behavior
- Team role `Originator` is inferred from the plan creator, not selected as a standard role option
- Deficiency creation is only allowed from checklist questions answered `No` during the Execution phase
- Production data seeding is not included in the current scaffold
- Some lifecycle gates or approval blocking rules may not be implemented in the current scaffold unless explicitly confirmed in lifecycle documentation and code
